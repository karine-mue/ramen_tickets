# 要件定義書
## ラーメン店券売機・厨房連携システム

## 1. 目的

サービスエリア等に設置される、複数台構成可能なラーメン店向け券売機システムにおいて、券売機注文、決済、発券、厨房連携、呼出番号管理、障害復旧を一貫したトランザクションとして扱う。

防止対象は以下。

```text
- 入金済みだが注文が消失する
- 釣銭補充や再起動で取引状態が失われる
- 複数券売機で注文番号が衝突する
- 厨房への二重送信が発生する
- どのラーメンにどのトッピングを乗せるか不明になる
- 客向け番号と内部監査IDが混同される
```

---

## 2. システム構成

```text
[券売機 KIOSK-01] ┐
[券売機 KIOSK-02] ├─> [注文受付サーバ / Order Server] ─> [厨房キュー / Kitchen Queue]
[券売機 KIOSK-03] ┘                                      └> [厨房表示・印刷]
```

### 構成要素

```text
KIOSK:
  商品選択、注文構成、入金、取引ID発行、注文送信、食券発行

ORDER_SERVER:
  冪等受付、中央採番、注文状態管理、厨房キュー登録

KITCHEN:
  厨房表示、調理状態管理、完成・呼出管理

STAFF_CONSOLE:
  未完了取引確認、返金、再発券、厨房再送、手動処理
```

---

## 3. 基本概念

## 3.1 ID体系

```text
transaction_id:
  券売機単位の決済・冪等キー。
  例: 20260503-SA012-K02-000188

order_id:
  店舗全体の内部注文ID。
  決済、返金、監査、障害復旧用。
  例: 20260503-SA012-000427

display_no:
  客向け呼出番号。
  4桁。
  例: 0427
```

### 推奨桁数

```text
display_no:
  0001〜9999
  営業日単位でリセット

order_id:
  YYYYMMDD + STORE_ID + 6桁連番

transaction_id:
  YYYYMMDD + STORE_ID + KIOSK_ID + 6桁連番
```

---

## 3.2 注文構造

ラーメンは単純な明細行ではなく、1杯単位の `MealUnit` として扱う。

```text
Transaction
└─ Order
   ├─ MealUnit
   │  ├─ BaseItem: ラーメン本体
   │  ├─ Options: トッピング、大盛り、麺硬さ等
   │  └─ Exclusions: ねぎ抜き等
   │
   ├─ SideItem
   ├─ DrinkItem
   └─ AlcoholItem
```

### NG構造

```text
醤油ラーメン × 1
味噌ラーメン × 1
味玉 × 1
大盛り × 1
ねぎ抜き × 1
```

この構造では、味玉・大盛り・ねぎ抜きの適用先が不明になる。

### OK構造

```text
[1] 醤油ラーメン
    - 味玉
    - 大盛り
    - ねぎ抜き

[2] 味噌ラーメン
    - オプションなし
```

## 4. 機能要件

## 4.1 券売機注文機能

```text
FR-001 商品カテゴリを表示できること
FR-002 ラーメン本体を選択できること
FR-003 ラーメン1杯ごとにトッピング・大盛り・麺硬さ等を設定できること
FR-004 トッピング単体が親ラーメンなしで注文されないよう制御できること
FR-005 「全員分に適用」操作は内部的に各MealUnitへ展開すること
FR-006 サイドメニュー、飲料、酒類を同一Transaction内に含められること
FR-007 注文確認画面でMealUnit単位に内容確認できること
```

## 4.2 決済前チェック機能

入金受付前に、以下をチェックする。

```text
FR-101 釣銭在庫
FR-102 受付可能紙幣・硬貨
FR-103 券紙残量
FR-104 プリンタ状態
FR-105 注文受付サーバ接続状態
FR-106 厨房キュー受付可否
FR-107 商品在庫・販売可否
```

釣銭不足がある場合、入金を開始してはならない。

```text
注文確定
→ 決済前チェック
→ OKなら入金受付
→ NGなら販売不可または店員対応
```

## 4.3 決済機能

```text
FR-201 現金決済を受け付けること
FR-202 投入金額を永続化すること
FR-203 必要額到達後、注文確定処理へ進むこと
FR-204 入金後エラー時は未完了取引として保持すること
FR-205 返金可能な場合は自動返金すること
FR-206 返金不能な場合は店員対応へ遷移すること
```

## 4.4 発券・厨房連携機能

```text
FR-301 order_idを中央採番すること
FR-302 display_noを中央採番すること
FR-303 同一transaction_idの再送は同一order_id/display_noを返すこと
FR-304 厨房キューへ注文を登録すること
FR-305 厨房チケットはMealUnit単位で出力できること
FR-306 客向け控えにはTransaction全体の明細を出力できること
FR-307 厨房連携失敗時は再送または店員対応へ遷移すること
```

## 4.5 複数券売機対応

```text
FR-401 券売機はn台増設可能であること
FR-402 注文番号採番は中央サーバで行うこと
FR-403 各券売機がローカルにdisplay_noを採番しないこと
FR-404 transaction_idを冪等キーとして扱うこと
FR-405 通信断・応答不明時は同一transaction_idで再送すること
```

## 4.6 アルコール販売

```text
FR-501 酒類商品にalcohol属性を持たせること
FR-502 酒類を含む注文では年齢確認表示を行うこと
FR-503 運転者への販売不可表示を行うこと
FR-504 酒類点数を別カウントできること
FR-505 必要に応じて店員確認対象にできること
```

## 4.7 取引上限

```text
max_order_lines = 24
soft_warning_order_lines = 16
max_quantity_per_line = 9
max_total_quantity = 50
max_kitchen_work_units = 15.0
soft_warning_kitchen_work_units = 10.0
max_alcohol_units = 12
max_total_amount = 50000
```

### カウント単位

```text
order_lines_count:
  明細行数

item_quantity_total:
  商品数量合計

kitchen_work_units:
  厨房負荷

alcohol_units:
  酒類数量

ticket_print_lines:
  印字行数
```

## 4.8 保守・復旧機能

```text
FR-601 保守モード移行前に未完了取引を確認すること
FR-602 未完了取引がある場合、保守モードへ直行しないこと
FR-603 補充・扉開放・再起動で取引状態を破棄しないこと
FR-604 起動時に未完了取引をスキャンすること
FR-605 店員画面で再発券、返金済み登録、厨房再送、手動処理済み登録ができること
```

## 5. 非機能要件

## 5.1 永続化

以下の状態は揮発させてはならない。

```text
- 入金状態
- 投入金額
- 注文内容
- transaction_id
- order_id
- display_no
- 発券状態
- 厨房送信状態
- 返金状態
- 未完了取引状態
```

### 状態分類

```text
L0 View State:
  画面表示。消えてよい。

L1 Session State:
  操作中状態。未確定なら消えてよい場合がある。

L2 Committed State:
  ユーザーが確定した状態。消してはいけない。

L3 Scheduled/Obligated State:
  未来実行義務を持つ状態。永続化と復元が必要。
```

券売機の入金済み注文は L2。  
厨房送信待ち・再送待ちは L3。

## 5.2 冪等性

```text
NFR-101 同一transaction_idで複数回送信された注文は、同一注文として扱うこと
NFR-102 同一transaction_idで内容が異なる場合は不整合として拒否すること
NFR-103 応答不明時の再送で二重注文を発生させないこと
```

## 5.3 監査ログ

```text
NFR-201 取引開始
NFR-202 商品選択確定
NFR-203 入金開始
NFR-204 入金完了
NFR-205 注文送信
NFR-206 中央採番
NFR-207 厨房登録
NFR-208 発券
NFR-209 釣銭返却
NFR-210 エラー
NFR-211 店員介入
NFR-212 手動処理
```

をログとして保持すること。

---

## 6. 受入条件

```text
AC-001 入金済み取引が画面リセット・保守操作・再起動で消失しない
AC-002 釣銭不足時は入金前に販売停止できる
AC-003 複数券売機で注文番号が衝突しない
AC-004 通信断再送で二重注文が発生しない
AC-005 トッピングが必ず対象MealUnitに紐付く
AC-006 厨房表示がMealUnit単位で確認できる
AC-007 店員が未完了取引を復旧できる
AC-008 客向けdisplay_noと内部order_id/transaction_idが分離されている
```

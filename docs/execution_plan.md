# 実行計画書
## ラーメン店券売機・厨房連携システム

## 1. 方針

券売機単体ではなく、以下の単位でPoCを行う。

```text
1. 注文データモデル
2. MealUnitとトッピング制御
3. 決済状態の永続化
4. 冪等注文受付
5. 中央採番
6. 厨房キュー
7. 障害復旧
```

UIより先に、状態とIDを固める。

---

## 2. フェーズ構成

```text

- 厨房表示か紙印刷か

- 酒類販売有無
- 店員介入フロー
- 返金フロー
- 釣銭補充手順
- 券紙補充手順
```

### 成果物

```text
- 現場業務フロー
- 商品カテゴリ一覧
- 障害対応パターン一覧
```

## 3. テスト計画

## 3.1 正常系

```text
TC-001 ラーメン1杯を注文し、現金決済し、発券される
TC-002 家族4人分のラーメンとサイドを1Transactionで注文できる
TC-003 ラーメンごとに異なるトッピングを設定できる
TC-004 酒類を含む注文で確認画面が出る
TC-005 複数券売機から同時注文して番号が衝突しない
```

## 3.2 異常系

```text
TC-101 釣銭不足時に入金受付へ進まない
TC-102 入金後に発券失敗しても取引が残る
TC-103 注文送信後、応答前に通信断しても再送で二重注文にならない
TC-104 釣銭補充で画面リセットしても取引が消えない
TC-105 電源断後、起動時に未完了取引が表示される
TC-106 同一transaction_idで別内容を送信した場合、拒否される
```

## 3.3 境界値

```text
TC-201 display_no 9999到達時
TC-202 24明細行到達時
TC-203 16明細行超過時の警告
TC-204 max_quantity_per_line = 9 到達時
TC-205 max_kitchen_work_units到達時
TC-206 酒類上限到達時
```

---

## 4. 受入条件

```text
AC-001 入金済み取引が画面リセット・保守操作・再起動で消失しない
AC-002 釣銭不足時は入金前に販売停止できる
AC-003 複数券売機で注文番号が衝突しない
AC-004 通信断再送で二重注文が発生しない
AC-005 トッピングが必ず対象MealUnitに紐付く
AC-006 厨房表示がMealUnit単位で確認できる
AC-007 店員が未完了取引を復旧できる
AC-008 客向けdisplay_noと内部order_id/transaction_idが分離されている
```

## 5. 最小実装スコープ

初期PoCでは以下までを対象にする。

```text
- 券売機1台
- 注文受付サーバ1台
- 厨房キュー1画面
- 現金決済はモック
- 発券はログ出力で代替
- MealUnit構造
- 中央採番
- 冪等送信
- 未完了取引復旧
```

本番前に追加する。

```text
- 複数券売機同時実行
- 実機決済連携
- 実プリンタ連携
- 釣銭機連携
- 保守モード
- 店員画面
- 酒類確認
- ピーク負荷試験
```

## 6. 実装優先順位

```text
P0:
  MealUnitデータモデル
  transaction_id/order_id/display_no設計
  入金済み状態の永続化
  冪等注文受付
  中央採番

P1:
  厨房キュー
  発券状態管理
  未完了取引復旧
  保守モード前チェック

P2:
  上限管理
  酒類確認
  kitchen_work_units
  ピーク予測
  POS/交通量データ連携
```

---

## 7. 要点

```text
券売機は入力端末。
注文受付サーバが直列化点。
厨房キューが調理制御点。
MealUnitがラーメン注文の最小意味単位。
transaction_idが冪等性の鍵。
display_noは客向け番号であって内部IDではない。
決済済み状態と未来実行義務は揮発させない。
```
